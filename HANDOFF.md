# 引き継ぎ指示書 — 都内ビュッフェ（食べログ点数3.4以上）一覧づくり

このドキュメントは、**ローカル環境の Claude Code** がこのタスクを引き継いで実行するための指示書です。
（作成: リモート環境の Claude Code / このリモート環境では食べログへの通信が遮断されており実行不可のため引き継ぎ）

---

## ゴール

食べログから、**都内（東京）のビュッフェ／バイキング**で **食べログ点数 3.4 以上**の店を抽出し、一覧表を作る。
さらに、**ユーザー本人のアカウントでログイン済みの状態**を使って、各店に **「行った / 未訪問」** を明記する。

最終成果物のイメージ（表）:

| # | 店名 | エリア | 食べログ点数 | ジャンル/特徴 | 行った/未訪問 | URL |
|---|------|--------|:---:|------|:---:|-----|
| 1 | 例）〇〇 | 品川 | 3.49 | ホテルブッフェ | 行った | https://tabelog.com/... |

出力形式は Markdown 表 + CSV の両方を推奨（`buffet_list.md` と `buffet_list.csv`）。

---

## なぜローカルでやるのか（前提）

- リモート実行環境ではネットワークポリシーにより `tabelog.com` への接続が**遮断**されていた（egress プロキシが 403）。ローカルにはこの制限がない。
- 「行った/未訪問」は**ユーザー個人の食べログアカウント**（マイページの「行ったお店」）にログインしないと分からない。
  ローカルなら、**ユーザーが普段使っている Chrome（食べログにログイン済み）のセッションをそのまま流用**できるため、
  **パスワードや認証情報を Claude に渡す必要がない**（＝安全）。

### ⚠️ 重要な注意（実行前にユーザーへ確認すること）
- 食べログの利用規約は**自動化アクセス（スクレイピング）を禁止**している。ボット検知でアカウントがロック/凍結される**リスク**がある。
- そのため **volume を抑え、ページ遷移ごとに待機（2〜5秒）**を入れ、短時間の大量アクセスを避ける。
- 可能なら**半手動**（ユーザーが画面を見ながら、Claude はページ内容の読み取り・整形に徹する）で進めるのが安全。
- 実行はユーザー本人の判断・自己責任で。開始前に一言確認を取ること。

## 手順

### ステップ0: 環境準備
- Node.js + Playwright（または Puppeteer）を用意。
- **ユーザーのログイン済み Chrome セッションを再利用する**方法を選ぶ（どちらか）:
  1. **CDP接続（推奨）**: ユーザーが Chrome を `--remote-debugging-port=9222` で起動（既存プロファイルで、食べログにログイン済みの状態）。
     Playwright から `chromium.connectOverCDP('http://localhost:9222')` で接続 → 既存のログインCookieをそのまま使える。
  2. **永続プロファイル**: `chromium.launchPersistentContext(userDataDir, {...})` でユーザーの Chrome プロファイルディレクトリを指定。
     ※Chrome を完全に終了してから使うこと（プロファイルロック回避）。
- どちらも **Claude がパスワードを入力しない**。既にログイン済みのセッションを借用するだけ。

### ステップ1: 「点数3.4以上のビュッフェ」リストの取得
食べログの検索/ランキングから収集する。候補URL:

- **バイキングランキング**: `https://tabelog.com/viking/tokyo/rank/`
- **絞り込み検索（点数順）**: `https://tabelog.com/tokyo/rstLst/buffet/?SrtT=rt&Srt=D`
  - `SrtT=rt` = 点数順、`Srt=D` = 降順。ページ送り（`.../2/`, `.../3/` …）で 点数が 3.4 を下回ったら打ち切り。
  - ログイン状態だと点数順ソートや詳細が安定して見える。
- 収集する項目: **店名 / エリア（駅・地名）/ 食べログ点数 / ジャンル / 店舗URL（rcid付き）**。
- **点数 3.4 以上のみ**を残す。点数は 3.xx 形式の数値（各店カードや店舗ページに表示）。
- セレクタ例（変わる可能性あり、実DOMを確認して調整）:
  - 店名: `a.list-rst__rst-name-target`
  - 点数: `.list-rst__rating-val` / `.c-rating__val`
  - 店舗リンク先URLから `rcid`（店舗ID）を控えておくと突き合わせに便利。

### ステップ2: ユーザーの「行ったお店」リストの取得
ログイン済みセッションで、マイページの**「行ったお店」**一覧を取得する。

- 入口: 食べログ右上のユーザーメニュー → マイページ →「行ったお店」。
  URL は概ね `https://tabelog.com/rvwr/<ユーザーID>/...` 系（実際にログインして辿って確定させる）。
- 取得項目: **訪問済み店舗の 店名 と rcid（店舗ID）** を全件（ページ送り含む）。
- これを「訪問済みセット」として保持。

### ステップ3: 突き合わせ（行った/未訪問の判定）
- ステップ1のビュッフェ各店について、**rcid が「訪問済みセット」に含まれれば「行った」**、なければ「未訪問」。
- rcid が取れない場合は**店名の正規化一致**（空白・記号・全半角を吸収）で補助的に照合。ただし同名別店の誤判定に注意。

### ステップ4: 出力
- `buffet_list.md`（Markdown表）と `buffet_list.csv` を生成。
- 列: `# / 店名 / エリア / 食べログ点数 / ジャンル・特徴 / 行った/未訪問 / URL`。
- 点数の高い順にソート。取得日時と件数、点数のしきい値(3.4)を冒頭に明記。

## 実装メモ / つまずきポイント
- **Cookie流用が肝**: 新規に Playwright を起動してログインし直すのではなく、既存ログイン済みセッションを借りる（CDP or 永続プロファイル）。これでログイン自動化＝規約違反リスクと credential 受け渡し問題の両方を回避。
- **待機を入れる**: `await page.waitForTimeout(2000〜5000)` を各ページ遷移に。連続高速アクセスは検知されやすい。
- **点数順ソートの仕様変更**: 食べログは仕様変更が多い。URLパラメータやDOMが想定と違ったら、実画面を見て調整。
- **ページ数上限**: 点数3.4は範囲が広く相当な件数になり得る。まず1〜数ページ試し、3.4を割る位置を確認してから全件回す。件数が多すぎる場合はユーザーにエリア（例: 23区内、特定の駅周辺）で絞るか相談。
- **途中経過を保存**: 取得のたびに逐次 CSV へ追記し、途中で止まっても再開できるように。

## このリモートセッションで確認済みの事実（引き継ぎ材料）
- `tabelog.com:443` はこの環境の egress ポリシーで **CONNECT 403**（遮断）。→ ローカルでは問題にならないはず。
- 参考: 食べログの東京バイキングランキング入口 `https://tabelog.com/viking/tokyo/rank/`、
  検索絞り込み `https://tabelog.com/tokyo/rstLst/buffet/?SrtT=rt`。
- Web検索で見えた例（未検証・要確認）: 品川プリンス「リュクス ダイニング ハプナ」≈3.49、両国のアパホテル内レストラン≈3.40 など。
  ※これらは検索スニペット由来で点数は変動する。必ず食べログ本体で最新値を確認すること。

## ローカルでの再開方法（ユーザー向け）
1. このブランチ（`claude/tabelog-tokyo-buffet-list-1n9pjz`）をローカルで `git pull`。
2. 食べログにログイン済みの Chrome を用意（CDP: `google-chrome --remote-debugging-port=9222 --user-data-dir=<既存プロファイル>` など）。
3. ローカルの Claude Code に「HANDOFF.md の手順で都内ビュッフェ点数3.4以上の一覧を作って。行った/未訪問も入れて」と依頼。
